UE Garbage Collection

沐云白 Lv3
  2025-07-27 11:14:31 2025-07-27 11:14:31 Created   2025-08-27 13:21:19 2025-08-27 13:21:19 Updated  

垃圾回收机制是通过反射实现的，而反射依赖于UPROPERTY()的标记和UObject，如果上层使用的UObject没有进行UPROPERTY()的标记，就可能会被错误回收，导致悬空指针。

垃圾回收原理可以看下面，这里主要讲限制对象的细节：

  • 不是UObject，就不会加入到垃圾回收系统中，需要用户自己手动管理。
  • 如果是UObject但是声明为裸指针且没有**UPROPERTY()**标记，就会导致被错误回收。

所以，垃圾回收的正确使用方式为：是UObject，然后用UPROPERTY()标记。

处理过程中的细节：

  • 记得加GC锁，防止出现多线程问题。

GC锁的目的是暂停其他线程防止GC的进行过程中，对象间的引用关系发生变化，导致出现问题。

  • 找到所有不可达的Object后先调用他们的BeginDestory()然后再析构。

优化算法

简易方案：

  1. 采用对象池。
  2. 调整调用的时机，原则上尽量少调用。
  3. 从根源上解决问题，少生成UObject。

标记不可达阶段的优化

用位翻转（翻转语义）进行优化。

原先进行标记不可达是需要遍历所有Obejct，然后将其在ReachableObjectFlag的值设为不可达，需要遍历所有Obejct，很消耗性能。

但是现在的处理方法直接将ReachableObjectFlag和MaybeUnreachableObjectFlag两个的标识符进行值交换，也就是进行语义翻转，也就是说，假设原本是标志第一位是ReachableObjectFlag、第二位是MaybeUnreachableObjectFlag，交换完就变成了第一位是MaybeUnreachableObjectFlag、第二位是ReachableObjectFlag，这样就可以直接实现不可达的标记效果。

但是能实现这个效果的前提就是一开始时所有Object都拥有可达标记，为了确保这个前提，就需要在创建UObject时进行标记可达的flag。

具体数据操作为，假设是0010 第二位是ReachableObjectFlag，最后一位是MaybeUnreachableObjectFlag，然后进行翻转，这样通过ReachableObjectFlag查找到的值就是0了，然后在进行可达标记时，设为0001，此时就回到了最初的状态，第二轮就可以重复进行。

对象聚类

当一个复杂的对象（如一个蓝图Actor）及其子对象（Components）被创建时，GC可以将它们组织成一个“集群”（Cluster）。这样，在进行可达性分析时，只需要检查集群的根对象。如果根对象可达，则整个集群中的所有对象都被视为可达，反之亦然。这减少了需要单独遍历的对象数量，提高了GC效率。

垃圾回收的伪实现：

  1. 启动垃圾回收，加锁，保持对象间的引用关系不变。
  2. 标记除了根对象、特殊对象的所有对象为“不可达”状态。
  3. 将根对象和特殊对象下的对象取消“不可达”状态的标记。
  4. 删除所有带有标记的对象。

GC问答

为什么只处理UObject？

  1. 因为GC是依赖于UObject中的反射实现的，需要GC就意味着需要加入UObject的一整套系统，这一套系统其实就是不小的开销。
  2. 并不是所有对象都适合 GC 处理，
    1. 比如底层的一些运算Vector等，这些如果拿个UObject包着的话就会非常庞大，消耗非常多。
    2. 不可预测性，系统自动的GC触发时机不可预测，无法确定对象何时被回收，所以对于底层细节，例如物理渲染这些就需要进行精细可控的控制。
  3. 为了提供**两套 ** 内存 管理机制，一套是UObject为核心的GC，另一套是手动处理内存的裸C++。
    1. 对于上层的gameplay系统，模块较多，引用复杂，如果手动回收则容易出现问题，使用GC机制可以帮助用户节省精力处理内存问题。
    2. 但是对于底层细节，会有更多的，如果使用垃圾回收可能会出现错误回收等问题。

以此方案将上层便利的逻辑开发和底层高效的可控的性能细节分离开，以此找到一个平衡点。

既然有了GC，那用户还需要进行内存管理吗？

需要的，

  1. GC并不是处理所有对象，他只处理UObject继承的对象，所以C++的基础类型或者第三方库的对象就需要手动处理。
  2. 有裸指针指向UObject的类型可能会被意外销毁，需要注意。
  3. 因为GC是通过引用链实现的，所以需要处理好类之间的引用关系，不然就会有些东西一直被引用所以无法销毁。
  4. GC其实也很耗性能的，需要进行一些控制，比如对象池等。

为什么GC依赖于反射？

因为在运行时，代码需要知道这个类中有什么对象，他才能往下遍历设置可达标记，而这个类中有什么对象的信息就是存储在反射系统中的UPROPERTY（）。

如果需要我自己实现垃圾回收机制我会怎么做？

首先我们要考虑垃圾回收的功能（需要实现上面功能）有什么，

①一个新的自定义newObject，用此来创建对象，但是不用delete与之对应。

②对象能在不需要使用的时候自动删除回收。

③不影响现在的功能。

为什么UE不采用引用计数的垃圾回收方式？

因为UE允许使用 UObject 的裸指针到处传递和赋值，而指针的传递和赋值底层没有办法捕捉这个操作，故引用计数法无法实施

并且引用计数需要重新的一套系统，而现在的GC可以顺便用上反射，方便很多。
