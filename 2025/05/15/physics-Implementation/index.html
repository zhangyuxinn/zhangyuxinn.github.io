Overview

Team2完成的honkai-battle-rail是一个基于自定义引擎完成的tank模拟战斗游戏，物理引擎再其中负责模拟游戏中各种需要真实的运动和碰撞行为，本文主要介绍项目中的物理引擎实现，包括游戏引擎需求，最终实现效果，物理引擎遇到问题，以及优化过程。

Requirements

2.1. Brief

物理引擎的更新包括以下过程：

1、ApplyExternalForce：

对物体施加外力，并将外力带来的线速度和角速度影响添加到物体的刚体组件上。

2、UpdateObjsStatus：

更新物体的状态，主要是通过物体的角速度和线速度应用积分运动x = x0 + vt求出物体的位置。

3、BroadPhase：

宽域碰撞检测，通过SAP算法计算场景中物体的aabb的碰撞，快速筛选出可能的碰撞对。

4、HandleCollision：

碰撞求解过程，将上一步求得的碰撞对进行碰撞求解，主要是计算出碰撞法线和碰撞深度，通过碰撞数据对碰撞的两个物体调整位置，施加冲量。

Existing Technology

  • Sweep And Prune（SAP）算法：从轴上快速筛选可能的aabb碰撞对。
  • GridMap：网格地图，AI寻路基于此地图的更新数据，在大量物体的情况可以用来加速射线检测。
  • Multi-Threads：多线程加速SAP算法，在大量物体时提升明显。
  • RayCast：射线检测，大量用于检测范围和Editor检测鼠标逻辑中。
  • AABB&MinMaxAABB：通过两种AABB类型适用于不同情况，两种类型可以相互转化。

Research

5.1 Physics Engine Design 物理引擎框架设计

在参考了多个主流引擎的的架构后，例如Box2D，PhysX等主流物理引擎，本引擎的框架设计重点在以下方面：

  • 模块化设计：将物理引擎的各种步骤解耦，方便后续维护和扩展。
  • 数据驱动：将物理引擎的各种计算放在物体的RigidBody组件中，并将场景中的碰撞体统一管理添加删除和更新。
  • 单独的时间步进：采用Fixed-Delta-Time来保证引擎的更新稳定性。
  • 引擎独立计算：只负责管理场景中需要物理引擎模拟的物体，不管理其他物体（即不带Collider和RigidBody的物体）。

5.2 Physics Engine Optimization 物理引擎优化

  • 优化点：本引擎的更新过程的四个步骤都有能存在优化的地方，但BroadPhase步骤提升是最明显的。
  • 优化概括：主要优化在于空间划分和SAP算法的优化，相比暴力两层循环提升明显。

5.3 Continues Collision Detection 连续碰撞检测

  • 原因：在高速运动场景中，通过连续碰撞检测来保证碰撞的正确性。
  • 方法：引擎主要是AABB碰撞盒，所以采用扫掠AABB来进行CCD碰撞检测。
  • 注意：引擎只是实现了CCD的接口，实际应用中并未开启CCD。

Implementation details

6.1 Data Structure Design 数据结构设计

物理引擎主要由PhysicsEngine类统一管理所有物理相关的物体逻辑，计算结果会返回到GameObject内部的组件中，完成更新后，物理引擎通过读取GameObject的数据来检测碰撞和计算碰撞结果。

需要注意的是，物理引擎是通过刚体来间接控制物体的，如果物体不带刚体或者碰撞体的话是不会收到物理引擎影响的。

6.2 Main Tick Logic 主要更新逻辑

6.2.1 ApplyExternalForce

遍历物体给所有接受重力的物体添加重力影响，同时将物体上一帧所受的外力/冲量应用到物体身上，改变物体的角速度和加速度。

6.2.2 UpdateObjsStatus

更新物体的状态，主要是通过物体的角速度和线速度应用积分运动x=x0+vt求出物体的位置以及应用阻力。该阶段是不会对Kinematic物体进行修改的。

6.2.3 BroadPhase

宽域碰撞检测，通过SAP算法计算场景中物体的aabb的碰撞，快速筛选出可能的碰撞对。该阶段是后续优化的主要阶段。

SAP****算法：

算法的思路是如果两个aabb重叠，那么这两个aabb在x和y上的投影也是重叠的，一个AABB投影在一个轴上就变成了一条直线，那么假如两个 AABB 相交，那么其在 x，y 上的投影出的两条直线必定会重叠。如果不相交，那么起码有一个轴上，两投影直线不重叠，因此，我们只需检测某条轴上投影相交的 AABB 对即可。Sweep and Prune 算法利用这个特性，将二维的 AABB 问题降到一维的直线问题。

①将所有 AABB 投影至特定轴上

②对轴上所有区间端点进行升序排序

③从小到大扫描投影轴

④遇到一个开始端点 s(i) ，将 s(i) 所属的 AABB(i) 与 L 中的所有 s 所属的 AABB 进行相交检测, 若发生相交就将碰撞添加到碰撞对列表中。

//选取方差较大的轴，意味着分布越分散的轴算法优化越好（更快排除不可能碰撞的物体），此游戏中在y轴上的碰撞和投影并不关键

//如果选取y轴甚至会退化到O（n^2)的复杂度，并且游戏中大部分物体是不移动的，所以在x和z之间选取。

可优化的过程包括排序，以及计算碰撞对等。

6.2.4 HandleCollision

需要注意的是该过程是将传统物理引擎的NarrowPhase和Solver阶段整合在一起，因为引擎现阶段并未扩展其他碰撞盒。

碰撞求解过程，将上一步求得的碰撞对进行碰撞求解，主要是计算出碰撞法线和碰撞深度，通过碰撞数据对碰撞的两个物体调整位置，施加冲量。

该阶段主要是调整两物体碰撞后位置，然后根据重量分配给予物体冲量，含有切向摩擦力调整。

注意点在于：

1、在调整位置的时候不要一次调整物体，一次性调整物体会导致质量差距较大的两个物体发生瞬移的现象。但也不要每次调整的位移过少，会导致穿透和粘连。

2、该部分还需要重构来完善，主要是在NarrowPhase的时候直接计算的是AABB的重叠区间，碰撞法线只会出现在三个对称轴，在需要真实模拟的时候无法达到预期效果，取质心连线可以模拟更真实的效果，但是会导致物体的移动不符合碰撞盒效果。

3、该阶段可以通过迭代多次趋近于最终结果，迭代次数过多会导致物体瞬移以及性能问题，过少又达不到多物体碰撞的真实效果，这里的迭代次数是3。
